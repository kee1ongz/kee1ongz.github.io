Unleashing your inner detective: A hilarious guide to becoming an Nmap-probe master and uncovering Web apps (with Portainer as our unsuspecting victim)

Created2023-03-30|Updated2023-03-30
|Post View:

0 Abstract

Information gathering is a critical step in the penetration process. One of the most critical steps is network sniffing/fingerprinting collection.

However, the return message/port number varies for different web applications.

This makes it possible that sniffing tools such as Nmap may not give correct feedback. (After all, they can’t all be included :/)

In this post, we take sniffing a container management tool as an example to illustrate how to analyze web applications that do not include fingerprints, and write Nmap signature fingerprint archives so you can use it for next analysis.

1 Introduction to Nmap

👁NMAP (Network Mapper) is an open source tool for network probing and pentesting. It is designed to quickly scan large networks/single target machines. Nmap uses raw IP messages to sniff network hosts, and information about the services/telegram types/firewalls they provide.

–Source: their website.

Nmap uses the fingerprint information from nmap-service-probes to analyze. The sniffer content is revealed back by sniffing back to —- to compare signatures.

However, for the fingerprints that are not included, nmap can only give a return message. It is left to the user to analyze manually. 🙃

In this example below, Nmap identifies ssh at port 22 and Mysql at port 3306, but only gives a return message for the service I started on port 9000 of my server.

So how to make the message recognizable to Nmap?

3 Adding fingerprints to Nmap-probe

Now let’s try to find how to add fingerprint information to Nmap’s probe.

Nmap’s service fingerprint format is simple and allows users to quickly write new signatures to detect new services: services can also include additional information about the device. For example, whether it has WAF, what language it is based on, the running version number, etc.

The probe needs to contain at least the information in the following table:

Directive Description
Exclude Ports excluded from probing
Probe The name of the protocol/application + the probe data to be sent
match Hard match, specifying the response message that matches the requirements
softmatch Similar to soft match, allows the scan to continue matching other lines
ports and sslports Define when to perform a probe
totalwaitms Response timeout setting
tcpwrappedms Used only for NULL probing to identify tcpwrapped (which can be considered tcp-based) services
rarity Describe how common the service is
fallback Defines which probe to use for backtracking when there is no match. For example, backtracking to GetRequest, GenericLines

As a classic example, take a look at how Nmaps’ NULL probe wroks. This is probably the easiest one in Nmap - it will only perform a simple banner crawler. It will not send any data;

In other words, it will only connect to the port - listen for a response - try to match something, and if it doesn’t work, just fxck off:

From this, we found that a probe can have multiple match lines and softmatch lines to detect services that respond to the same request data.

For the simplest service fingerprinting, just implement the following command: probe, rarity, ports, match .

How did this demo generate?

Referring to the description of probes in the official documentation of nmap, we can quickly learn the usage of these commands:

**1.probe **

  • <protocol> : TCP or UDP either
  • <probename>: user-defined to simply describe the probe name
  • <probestring>: Specifies the content to be sent by Nmap. The format is q|<data>|, note that some of the symbols need to be escaped.

**2.rarity **

rarity corresponds to the frequency at which this probe returns useful results. The higher the number, the less probes are considered.

Development details can be found in: https://nmap.org/book/vscan-technique.html#vscan-selection-and-rarity

**3.ports **

This line tells Nmap on which ports the services identified by this probe are typically found. This usage is similar to nmap -p.

**4.match/softmatch **

The most important command that determines whether your probes can identify the target successfully.

  • <service>: The name of the service whose pattern matches. For example ssh, smtp, http.

  • <pattern> The syntax of the pattern is similar to Perl: m/[regex]/[opts]. m is the start of the matching string, regex is the regular expression based on PCRE, and opts is the option part of the pattern. /i is the only one supported for now.

  • <versioninfo>:an optional parameter, including version number, os and other information. Refer to the following table for details:

For more information on the usage of probe, you can find them on the official documentation at https://nmap.org/book/vscan.html。
